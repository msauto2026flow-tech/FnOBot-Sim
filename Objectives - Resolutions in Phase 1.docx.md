STEP 1 — Review of your Trade Objective document + improvements
Your objectives document is a solid trading-first specification. It correctly prioritizes what matters — trend identification through OI + candles, futures volume for real indicators, and stock-level analysis beyond just indices. Here are my inputs on each point before we proceed:
Objective 1 (Trend from candles + OI every 3 min): Sound approach. One thing to add — you should define what "trend" means in measurable terms. Right now, the bot classifies trend as TRENDING_UP / TRENDING_DOWN / RANGING using HH/LL structure, but your objective implies you also want candle pattern recognition (engulfing, pin bars, inside bars). The v5 codebase does not do candlestick pattern detection — it only does Supertrend, EMA, and RSI on the 15-min frame. If you want pattern-based entries, that's a separate module to build.
Objective 2 (Futures data including volume): This is already partially implemented — fetch_futures() gets LTP and computes basis, and VWAP patches index candles with futures volume. But the futures OHLCV candles themselves are not stored anywhere. You're fetching them to extract volume, then discarding the price data. If you want to prepare a futures candle chart, you need to persist the raw 3-min futures candles to Excel or a separate data store.
Objective 3 (Indicators — VWAP, Supertrend, EMA, RSI): All four are implemented in indicators.py. However, your objective says "VWAP from 3-min candles" but the current implementation fetches 5-min candles for VWAP. Since you've moved to a 3-min scan interval, you should align the VWAP candle interval to 3-min as well for consistency. Also, the RSI divergence detector exists but it's swing-pivot based — it requires multiple 15-min candles to form pivots, so it won't fire in the first 90 minutes of trading. This is a known limitation worth noting.
Objective 4 (Nifty 50 weightage correlation): This is entirely missing from both the v5 action plan and the current codebase. The bot fetches Nifty 50 stock prices and shows gainers/losers, but it does not know each stock's index weightage, does not compute the weighted contribution of each stock to the index move, and cannot identify which stocks are dragging or lifting the index. This is a new module to build. The data is available — NSE publishes Nifty 50 weights quarterly, and you could hardcode or fetch them. The analysis would be: for each stock, compute stock_Chg% × weight% to get its index point contribution, then rank by contribution to see who's actually moving the index.
Objective 5 (Nifty 50 stock OI, strike prices, trends): This is the most ambitious and least developed requirement. Currently the bot only fetches stock spot prices from NSE equity. It does not fetch stock F&O data at all — no stock option chains, no stock OI, no stock futures. Kite Connect supports this (all NFO stocks are available), but the instrumentation and API call volume would be significant. For 50 stocks × ~20 strikes each × CE+PE, you're looking at 2,000+ quote calls per scan. At 3-min intervals, this will hit Kite's rate limits. My recommendation: don't fetch all 50 stocks' F&O data. Instead, build a screener that first identifies the top 5–10 "interesting" stocks (based on spot price movement, unusual volume, or heaviest Nifty weight movers), and then fetch detailed OI data only for those. This keeps the API load manageable and focuses on actionable opportunities.
Objective 6 (Storage — Excel locking problem): This is a real operational pain point you've identified. When Excel has the file open, Python's openpyxl.save() fails with a permission error. There are three solutions: (a) write to a temporary file and rename on close (atomic write), (b) migrate to SQLite which handles concurrent access natively, or (c) use a copy-on-write pattern where the bot writes to MarketData_YYYY-MM-DD_live.xlsx and copies to the main file only when the user closes it. For the 3-min vs 15-min computation question: fetch and store data every 3 minutes, but run the 15-min indicator calculations only when the scan timestamp aligns to a 15-min boundary (09:15, 09:30, 09:45, etc.). Between those, carry forward the last computed indicator values.
Objective 7 (Dashboard — HTML vs alternatives): HTML is the right choice for now. It's zero-dependency, opens in any browser, and your existing dark theme looks professional. For future retrospection, the better path is: keep HTML for live dashboard, but add a Streamlit or Dash app for historical analysis. Streamlit gives you interactive charts with date pickers, slider filters, and drill-down into past days' data — all from the Excel/SQLite store. Don't switch your live dashboard away from HTML though; its simplicity is a strength during market hours.
Objective 8 (Dashboard requirements): Mostly covered in v5 already. The specific gaps I see against your list: VIX is missing (not fetched), Option Theta value is not computed (you'd need to extend the Black-Scholes module to output Greeks, not just IV), the option chain doesn't have a "Next Expiry" tab (only nearest expiry is fetched), and the 5-min OI delta column doesn't exist yet (you have 3-min delta from the scan interval, and a reconstructed 1-hour delta, but no explicit 5-min).
Your phasing plan (Phase 1: Indices → Phase 2: Stocks): Correct approach. Build the index infrastructure rock-solid first, then extend to stocks. The stock search bar with autofill is a frontend feature that's straightforward in HTML/JS — a <datalist> element with the 50 symbols works natively without any library.
One missing objective I'd add: You don't mention position tracking or P&L monitoring anywhere. Once you're trading based on signals, you'll want the bot to track your open positions (Kite has an orders/positions API), compute real-time P&L, and enforce your two-strikes rule automatically. This should be Objective 9.
